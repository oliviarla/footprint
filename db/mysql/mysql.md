# MySQL 엔진

## 스레드 구조

* 스레드 캐시
  * 일정 개수만큼의 대기 스레드를 유지하여 필요 시 스레드를 바로 할당한다.
* Foreground Thread
  * MySQL 서버에 접속한 클라이언트 수 이상만큼 존재한다.
  * 주로 클라이언트의 쿼리를 처리하는 역할을 담당한다.
  * InnoDB 엔진을 사용하는 경우 데이터 버퍼나 캐시 작업을 포그라운드 스레드에서 진행하고, 버퍼로부터 디스크에 데이터를 기록하는 일은 백그라운드 스레드가 담당하게 된다.
* Background Thread
  * InnoDB의 경우 다음 작업들을 각자의 스레드에서 수행하게 된다.
    * Insert Buffer를 병합
    * 로그를 디스크로 기록
    * InnoDB 버퍼 풀 데이터를 디스크에 기록
    * 데이터를 버퍼로 읽기
    * 락이나 데드락을 모니터링

## 메모리 할당 및 사용 구조

* 글로벌 메모리 영역
  * 테이블 캐시, InnoDB 버퍼 풀, InnoDB Adaptive Hash Index, InnoDB 리두 로그 버퍼 등이 이 영역에 해당된다.
  * 모든 스레드에 의해 공유된다.
* 로컬 메모리 영역
  * 클라이언트 스레드가 쿼리를 처리하는 데 사용하는 영역이다.
  * 각 스레드마다 독립적으로 할당되며 공유되지 않는다.
  * 정렬 버퍼, 조인 버퍼, 바이너리 로그 캐시, 네트워크 버퍼 등이 이 영역에 해당된다.
  * 쿼리를 실행할 때에만 할당되었다가 해제되거나, 커넥션이 열려있는 동안만 할당되는 등 필요한 경우에만 메모리가 할당될 수 있다.

## 플러그인 스토리지 엔진 모델

* 스토리지 엔진을 비롯하여 다양한 기능을 사용하기 위해 플러그인을 설치하고 포함시킬 수 있다.
* MySQL 엔진과 스토리지 엔진은 핸들러를 통해 요청을 주고 받는다.
* 플러그인끼리는 통신이 불가능하고 MySQL 서버의 변수나 함수를 직접 호출하므로 안전하지 않다. 이로 인해, 컴포넌트라는 아키텍처를 대신해서 사용할 수 있다.

## 쿼리 실행 구조

1. 쿼리 파서
   * 쿼리 문장을 토큰으로 분리해 트리 형태의 구조로 만든다.
   * 이 과정에서 문법 오류를 발견할 수 있다.
2. 전처리기
   * 파서 트리를 기반으로 쿼리 문장에 구조적인 문제점이 있는지 확인한다.
   * 문장을 객체와 매핑하여 실제 존재하지 않거나 권한 상 사용할 수 없는 객체에 대한 쿼리를 발견할 수 있다.
3. 옵티마이저
   * 쿼리 문장을 가장 저렴한 비용으로 빠르게 처리할 지를 결정한다.
4. 실행 엔진
   * 실행 계획대로 핸들러에 작업을 요청하고 응답을 받아 처리한다.
5. 핸들러
   * 실행 엔진의 요청에 따라 데이터를 디스크에 저장하거나 디스크로부터 읽는 역할을 한다.

## 트랜잭션 지원 메타데이터

* 테이블 구조 정보나 스토어드 프로그램의 코드 정보 등 MySQL 서버 작동을 위한 데이터를 저장한 테이블들을 시스템 테이블이라고 하며, InnoDB의 테이블에 저장한다.
* 시스템 테이블과 데이터 딕셔너리 정보를 `mysql.ibd` 파일에 저장한다.
* InnoDB 스토리지 엔진 이외의 스토리지 엔진들의 메타데이터는 SDI(Serialized Dictionary Information) 파일에 저장된다.
