# 명시적 락

## Lock과 ReentrantLock

* 자바의 Lock 인터페이스에는 여러 락 관련 기능에 대한 추상 메서드를 정의한다.
* 암묵적인 락(synchronized)과 달리 조건없는 락, 폴링 락, 타임아웃이 있는 락, 대기 상태에 인터럽트를 걸 수 있는 락이 포함되어 있다.
* 암묵적인 락 외에 명시적인 락 클래스가 필요한 이유는 다음과 같다.
  * 락을 확보하고자 대기하고 있는 상태의 스레드에 인터럽트를 걸 수 있다.
  * 블록 구조를 사용하지 않고 락을 걸 수 있다.
* 항상 락을 사용할 때 try-finally를 통해 어떤 일이 있더라도 락을 해제해야 한다.

```java
Lock lock = new ReentrantLock();

// ...
lock.lock();
try {
  // ...
} finally {
  lock.unlock();
}
```

#### 폴링과 시간 제한이 있는 락

* tryLock 메서드를 사용하면 락을 획득 시도하느라 프로그램이 멈춰버리는 상황을 방지할 수 있다.
* 락 확보를 위한 시간 제한을 두기 위해 인자를 지정하거나 폴링 방법을 사용하면 락을 확보하지 못하더라도 통제권을 다시 얻어 재시도할 수 있다.
* 지정된 시간 내에 끝나야 하는 작업에서 락을 획득해야 하는 상황이 있다면, 타임아웃 값을 지정해서 특정 시간 내에 획득하지 못하면 작업을 종료하게 만들 수 있다.

#### 인터럽트 걸 수 있는 락

* 작업 도중 취소시킬 수 있어야 하는 작업을 수행하는 경우 락 확보 시도 중에 인터럽트를 걸어 멈춘 후 취소 처리를 할 수 있다.

#### 블록을 벗어나는 구조의 락

* 암묵적 락을 사용하면, 락을 확보하고 해제하는 부분이 블록 구조에 한정되어 있고 블록에서 벗어나는 순간 락을 자동으로 해제한다.
* 락 스트라이핑 등을 제공하기 위해 복잡한 구조의 프로그램에 락을 적용하려면 블록 구조가 제한적일 수 있다.
* 명시적 락을 사용하면 유연하게 락을 획득/해제할 수 있다.

## 공정성

* ReentrantLock에서는 요청한 순서를 지키며 락을 획득하게 하는 **공정 락**과 대기하는 큐에 대기중인 스레드가 있더라도 이를 넘어서 락을 확보할 수 있는 **불공정 락**을 제공한다.
* 공정하게만 처리하는 것보다는 통계적인 공정함, 즉 대기중인 스레드는 언젠가 락을 확보한다라는 기능 정도면 충분하고 성능 저하도 적다.
* 대기 상태에 있는 스레드를 실행 상태로 돌리고 실제로 실행시키는 데에 시간이 걸리기 때문에, 공정 락이 비공정 락보다 느릴 수밖에 없다.
* 락을 확보하고 사용하는 시간이 상대적으로 길거나 락 요청이 발생하는 시간 간격이 긴 경우 공정 락이 유리하다.

## ReadWriteLock

* 대부분의 데이터는 내용을 변경할 수 있으며 종종 변경되지만, 대다수의 작업은 읽기 작업이다.
* 읽기 연산은 여러 스레드에서 동시에 실행할 수 있도록 하고 쓰기 연산은 한 스레드만 동작할 수 있도록 할 수 있다.
* 최적화된 상황이 아닌 곳에 적용하면 일반 락에 비해 성능이 떨어질 수 있다.

#### ReentrantReadWriteLock

* 읽기 락과 쓰기 락에 재진입 가능한 락 기능을 제공한다.
* 공정 락을 사용하는 경우 대기 큐에서 대기한 시간이 가장 긴 스레드에 우선권을 주며, 불공정 락을 사용하는 경우 어떤 스레드가 락을 확보할지 알 수 없다.
* 쓰기 락을 확보한 상태에서 읽기 락을 확보할 수 있다. (다운그레이드 가능)
* 읽기 락을 확보한 상태에서 쓰기 락은 확보할 수 없다. (업그레이드 불가능, 데드락 발생 방지)
* 쓰기 락은 확보한 스레드만이 해제할 수 있다.
* 읽기 락은 여러 스레드가 동시에 확보 가능한 세마포어이며, 확보중인 스레드를 추적 가능하다.
