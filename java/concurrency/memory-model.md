# 자바 메모리 모델



## 자바 메모리 모델 (JMM)

* 자바 메모리모델은 변수에 저장된 값이 어느 시점부터 다른 스레드의 가시권에 들어가는지 JVM이 해야 할 최소한의 보장을 한다.
* 높은 성능의 JVM을 다양한 종류의 프로세서 구조에서 동작하게 해야 하는 요구사항을 쉽게 구현하기 위해 설계되었다.
* 프로그램 내 모든 작업을 대상으로 미리 발생이라는 부분 재배치 연산을 정의한다.
  * 미리 발생 현상에 대한 규칙은 대표적으로 다음과 같은 것들이 있다.
    * volatile 변수의 경우 쓰기 작업이 모든 읽기 작업보다 미리 발생한다.
    * Thread.start 작업은 스레드의 모든 작업보다 미리 발생하며, Thread.join 메서드가 리턴되기 전 스레드의 모든 작업이 미리 발생한다.

### 플랫폼 메모리 모델

* 메모리를 공유하는 멀티 프로세서는 각 프로세서 안에 캐시 메모리를 가진다. 캐시 메모리는 주기적으로 메인 메모리와 동기화된다.
* 각 프로세서 아키텍처는 캐시 일관성을 저마다 다르게 지원한다. 일시적으로 일관성을 희생하고 성능을 높이는 경우도 많다. 즉, 동일한 순간에 같은 메모리 위치에서 각 프로세서가 서로 다른 값을 읽을 수 있다.
* JMM, 프로세서들은 프로그램의 코드가 실행되는 방법이 단 한 가지이며 어떤 프로세서이든 맨 마지막에 설정한 값을 읽을 수 있는 **순차적 일관성**을 지원하지 않는다.
* JVM에서는 JMM과 그 기반이 되는 하드웨어 메모리 모델의 차이를 메모리 배리어를 적절히 활용해 보완한다.
* 순차적 일관성을 가지고 있어 프로그램 내부의 모든 작업이 고정된 전역 순서에 따라 실행되는 프로그램은 올바르게 동기화된 프로그램이다.

### 재배치

* 특정 작업이 지연되거나 예측과 다른 순서로 실행되는 것 처럼 보이는 문제를 재배치라고 한다.
* 제대로 동기화되지 않은 프로그램에서 스케줄러가 작업을 겹쳐 실행하면 이러한 문제가 발생할 수 있다.
* 각 스레드의 내부에서 일어나는 작업은 다른 스레드와의 연결 관계가 없다.
* 동기화가 잘 된 상태에서는 컴파일러, 런타임, 하드웨어 모두 JMM이 보장하는 가시성 수준을 위반하는 재배치를 발생시키지 않는다.

### 동기화 피기백

* 락으로 보호되어 있지 않은 변수에 접근하고 사용하는 순서를 정의할 때 별도의 락을 사용하지 않고 모니터 락이나 volatile 변수 규칙 등을 통해 미리 발생 규칙을 정의할 수 있다.
* 명령이 나열된 순서에 민감하며 오류가 발생하기 쉽다.
* 다른 목적으로 만들어 사용하고 있는 미리 발생 순서 규칙을 객체의 가시성 확보하는 데에 함께 사용하고 있으므로 피기백이라고 부른다.

## 안전한 공개

* 공유된 참조를 공개하는 작업이 다른 스레드에서 해당 참조를 읽는 것보다 미리 발생해야 한다.&#x20;
  * 예를 들어 객체에 대한 참조에 값을 쓰는 작업과 객체 내부의 변수에 값을 쓰는 과정에서 재배치가 일어나면, 객체 참조는 최신 참조 값을 사용하지만 객체 내부의 변수는 아직 쓰기 작업이 끝나지 않은 값일 수 있다.
* 늦은 초기화 방식으로 싱글톤 객체를 생성할 때에도 객체를 공개하는 부분에서 동기화하지 않으면, 부분 구성된 객체를 참조하게 될 수 있다.

```java
public class UnsafeLazyInitialization {
    private static Resource resource;

    public static Resource getInstance() {
        if (resource == null) { //
            resource = new Resource(); // Race condition happens here
        }
        return resource;
    }
}
```

* 따라서, 특정 객체를 공개하는 일이 그 객체를 사용하려는 작업보다 미리 발생하도록 구성되어 있어야 한다.
* 락으로 보호되어 있는 공유 변수나 공유된 volatile 변수를 사용할 때는 읽기/쓰기 작업에 대한 미리 발생 관계가 항상 보장되어 있다.
* 미리 발생 관계가 보장되는 것은 안전한 공개에 의해 보장되는 가시성과 실행 순서보다 강력하다.
* 미리 발생 규칙은 메모리 작업 수준에서 일어나는 순서의 문제를 다루며, 안전한 공개 기법은 코드 작성과 비슷한 수준에서 동작하는 동기화 기법이라는 차이가 있다.

## 초기화 안전성

* 초기화 안전성을 확보하면, 올바르게 생성된 불변 객체는 어떤 방법으로 공개하더라도 여러 스레드에서 동기화 구문 없이 안전하게 사용 가능하다.
* final로 선언된 변수를 가지는 클래스는 초기화 안전성 조건 덕분에 해당 객체 참조를 최초로 생성하는 과정에서 재배치가 발생하지 않는다. 단, final로 선언되지 않은 변수가 함께 있거나 다른 메서드에서 final 변수의 내용을 변경(Set 같은 자료구조를 사용하는 경우 이에 해당된다.)할 수 있다면 초기화 안전성이 깨지게 된다.
