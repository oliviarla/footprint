# 병렬 프로그래밍 테스트

## 정확성 테스트

* 정확성 테스트에서는 어떤 변수가 올바른 값을 정확히 알고 있는지, 최종적으로 어떤 값을 가져야 하는지 확인해야 한다.
* 테스트 프로그램 작성 시에는 테스트 설계 과정에서 놓친 기능 명세를 찾아가는 과정이 포함된다.

### 대기 상태에 들어가는 메서드 테스트

* 대기 상태에 들어갈 것으로 예상되는 메서드를 테스트하는 방식은 다음과 같다.
  * 대기 상태에 빠지게 만들어주는 메서드를 호출하는 별도 스레드를 둔다.
  * 스레드가 대기 상태에 들어가면 Interrupt를 걸어 InterruptedException이 발생하였는지 확인한다.
  * 원하는 연산을 처리했는지도 검증한다.
* Thread#getState 메서드는 스레드의 대기 상태를 확인하기에 적합하지 않다.
  * JVM 구현 방법에 따라 스핀 대기 방식을 사용하기 때문에 대기 상태라고 해서 항상 WAITING, TIMED\_WAITING 상태가 되는게 아니라 RUNNING 상태로 전환될 수도 있기 때문이다.
  * 대기 상태에 진입했더라도 대기 상태에 안정적으로 자리 잡기까지 시간이 걸릴 수 있다.
  * 디버깅 정보를 얻어내는 용도로 사용하는 것이 좋다.

### 안전성 테스트

* 스레드 간 공유 가능한 데이터를 사용하는 과정에서 발생하는 오류를 테스트하려면 여러 스레드를 충분히 동작시키고 테스트 대상 클래스의 상태가 적절한지 검증해야 한다.
* 실제 데이터를 관리할 때 이와 완벽하게 동일한 내용을 담는 자료구조를 만들어 함께 저장하는 방식을 사용하면 모든 항목을 그대로 뽑아내 검증할 수 있다. 하지만 자료구조에 항목을 추가하고 제거하는 과정에서 스레드 동기화 작업이 필요하다.
* 스레드를 많이 돌릴수록 실행 중인 스레드와 대기 상태에 들어간 스레드가 교차하고 상호 작용이 발생하는 경우가 많아지므로 오류를 찾을 확률이 높아진다.
* CPU 프로세서의 개수, 운영체제, 프로세스 아키텍처 등을 다양하게 변경하면서 테스트하면 특정 시스템에서만 발생하는 오류를 찾아낼 수도 있다.

#### 체크섬 테스트

*   입출력 데이터에 대한 체크섬을 구해 저장하고, 체크섬 목록과 실제 데이터의 목록에서 체크섬을 구해 비교하고 검증하는 방법도 있다. 체크섬은 객체의 해시코드 값과 nanoTime을 이용해 xorShift한 랜덤 값을 사용하는 정도면 충분하다.

    ```java
    static int xorShift(int y) {
        y ^= (y << 6);
        y ^= (y >>> 21);
        y ^= (y << 7);
        return y;
    }
    ```
* 체크섬을 각 스레드마다 유지시키면 동기화할 필요가 없으므로 원하는 테스트 시나리오에만 집중할 수 있다.

#### 스레드를 항상 병렬로 실행시키기

* 각 스레드를 생성/실행하거나 종료하는 과정에 많은 부하가 걸리는 경우, 여러 스레드가 함께 병렬로 실행되는 구간이 줄어든다.
* 이를 해결하려면 CountDownLatch를 사용해 모든 스레드가 준비되어 Latch가 0이 될 때까지 대기하고 실제 작업을 시작하거나, CyclicBarrier를 사용해 전체 작업 스레드 개수 + 1 만큼 초기화해두고 대기한 후 동시에 작업을 시작할 수 있다.

#### 자원 관리 테스트

* 메모리를 비롯한 다양한 자원을 유출하는 일이 없는지 테스트해야 한다.
* 메모리 상황을 파악하기 위해 힙 조사 도구를 사용하여 GC를 강제로 실행한 후 메모리 정리가 되는지 확인할 수 있다.
* 정확한 테스트를 위해 사용하던 참조 객체를 null로 세팅해야 할 수도 있다.

#### 스레드 교차 실행량 확대

* 공유 자원을 사용하는 부분에서 Thread#yield 메서드를 호출해 컨텍스트 스위치가 많이 발생하도록 유도할 수 있다.
* 예를 들어 계좌 이체의 상황에서 A 계좌에서 인출하고 B 계좌에 입금한다면, A 계좌에서 인출한 후 Thread#yield 메서드를 호출해 잠시 멈춘 후 B 계좌에 입금할 수 있다.
* Thread#yield 메서드를 호출하는 코드는 AOP를 통해 관리하면 간편하다.

## 성능 테스트

* 성능 테스트는 다음 두 가지 목적을 가진다.
  * 특정 환경에서의 시나리오를 정해두고 시나리오를 통과하는 데에 얼마나 걸리는지 측정하는 것
  * 성능과 관련된 스레드 개수, 버퍼 크기 등 다양한 수치를 뽑아내어 성능을 최대화하는 것
* 스레드 개수를 늘리더라도 대기 상태에 들어갔다 나오면서 동기화 작업을 하면 성능 향상이 발생하지 않을 수 있다.
* 테스트 프로그램이 실행되는 동안 perfbar 등으로 CPU 성능을 모니터링할 수 있다.
* 처리량 혹은 단일 작업을 처리하는 데 걸리는 시간을 주로 측정한다.
* 단일 작업 처리에 걸리는 시간을 측정하는 경우 variance를 중요한 수치로 간주하는 경우가 있다. 예측성이 중요한 성능 지표인 경우 Quality of Service 결과가 중요하다.
* 개별 작업을 처리하는 속도가 컴퓨터가 갖고 있는 시간 측정의 최소 단위와 비슷할 정도로 아주 빠르면 통계 값을 제대로 구하기 어려우므로, 여러 개별 작업을 묶어 일괄 처리하는데 걸린 시간을 측정하는 것이 좋다.&#x20;

## 성능 측정의 함정

#### 가비지 컬렉션

* 성능 측정 작업 중 가비지 컬렉션이 진행되는 경우 테스트의 실행 시간에 크게 영향을 미칠 수 있다.
* 이를 해결하기 위해 1. 테스트 도중 가비지 컬렉션이 발생하지 않도록 하거나 2. 테스트 결과에 객체 생성 혹은 가비지 컬렉션이 발생하는 것을 자연스럽게 적절히 반영할 수 있다. 2번째 방법을 사용하는 경우 테스트 프로그램을 길게 실행할 수 있으며 실제 상황과 유사한 성능을 측정할 수 있다.

#### 동적 컴파일

* 최근 사용되는 JVM은 바이트코드 인터프리트 방식과 동적 컴파일 방식을 혼용해 사용한다.
* 따라서 컴파일 작업이 어느 시점에 실행되는지 정확히 파악하기 어렵다.
* 컴파일으로 인해 성능 측정이 올바르게 되지 않는 상황을 막기 위한 가장 간단한 방법은 테스트 프로그램을 오래 실행시켜 모두 컴파일 되어있도록 만드는 것이다.
* 시간을 측정하지 않는 웜업 테스트를 미리 실행시켜 필요한 코드를 컴파일시키고 이후 시간을 측정할 수 있다.
* 핫스팟 JVM을 사용하는 경우 `-XX:+PrintCompilation` 옵션으로 동적 컴파일 작업 시점에 메시지를 출력할 수 있다. 이를 통해 테스트 측정 시점을 추정할 수 있다.
* JVM에서 내부적으로 작업을 처리하기 위해 백그라운드 스레드를 사용하는데, 이 작업이 테스트에 영향이 가지 않도록 주기적으로 테스트 사이에 텀을 두어 작업이 처리되도록 하는 것이 좋다.

#### 의미 없는 코드 제거

* 테스트 시에 실제 상황과 다르게 의미 없는 코드를 실행하면 최적화 과정에서 제거될 수 있다.
* 병렬 프로그램에서는 공유 자원에 접근하여 처리할 작업을 뽑아낸 후 스레드 내부 데이터를 통해 작업하는 패턴이 일반적이다.
* 스레드 내부 데이터를 통해 작업하는 부분을 실제와 유사하게 작성하고 테스트해야 올바른 결과를 낼 수 있다.
* 이를 위해 프로그램 코드가 만들어낸 결과 값을 어딘가에서 사용하도록 해야 하며, 이 과정에서 추가적으로 동기화나 자원 소모가 없도록 주의해야 한다.
* 조건문에서 값을 사용하고, 조건이 절대 일치되지 않는 상황을 만든 후 내부에서 System.out.println() 호출하도록 하면 무의미한 코드로 제거당하지 않을 것이다.
