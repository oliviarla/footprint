# 동기화 클래스 구현

## 상태 종속성 관리

### 상태 종속적

* 락을 잡고 상태 변수를 확인한 후 조건에 부합하지 않는다면 락을 풀고 다시 대기하는 작업을 상태 종속적인 블로킹 작업이라고 한다.
* 상태 종속적인 메서드는 선행 조건과 관련한 오류가 발생하면 1. 예외를 발생시키거나 2. 오류 값을 리턴하거나 3. 원하는 조건에 도달할 때 까지 대기한다.
* 폴링하고 대기하는 반복 작업을 통해 원하는 상태가 될 때까지 대기하는 작업을 하는 경우 어느 간격으로 대기할 지에 따라 응답성과 CPU 소모 간의 트레이드 오프가 발생한다.

### 조건 큐

* 조건 큐를 이용하면 특정 조건이 만족할 때 까지 대기할 스레드들을 담아둘 수 있다.
* 자바의 모든 객체가 가지는 wait, notify, notifyAll 메서드는 암묵적인 조건 큐 API라고 볼 수 있다.
* wait 메서드는 현재 확보하고 있는 락을 자동으로 해제하면서 현재 스레드를 멈춰달라고 요청한다.
* 다른 스레드가 락을 확보해 객체 내부 상태를 바꾸고 notify 메서드를 호출하면, wait 메서드로 대기하던 스레드들이 깨어나게 된다. 이 때 wait 메서드는 리턴되기 전에 락을 확보한다.
* notify 메서드는 대기 상태에 들어가 있는 스레드 하나를 골라 대기 상태를 풀어준다.
* notifyAll 메서드는 대기 상태에 있는 모든 스레드를 풀어준다.
* notify, notifyAll 메서드를 호출하기 전 반드시 조건 큐 객체에 대한 락을 확보하고 있어야 하며, 호출이 완료되면 락을 풀어야 wait 메서드에서 락을 획득할 수 있다.

## 조건 큐 활용

### 조건 서술어

* 조건부 대기 기능을 동작하게 하는 데에 가장 중요한 요소이다.
* 특정 기능이 상태 종속적이 되도록 만드는 선행 조건을 의미한다.
* 예를 들어 버퍼에서 값을 꺼낼 때의 조건 서술어는 "버퍼에 값이 있어야 한다"이다.
* 조건 서술어는 클래스 내부의 상태 변수를 기반으로 하며, 상태 변수는 락으로 동기화되어 있으므로 조건 서술어를 만족하는지 확인하려면 락을 확보해야만 한다.
* 특정 조건 서술어를 놓고 wait 메서드를 호출할 때 호출자는 항상 해당하는 조건 큐에 대한 락을 확보한 상태여야 한다.

### 조건 서술어와 wait

* 하나의 조건 큐에 여러 조건 서술어를 연결해 사용할 수 있다.
* 따라서 wait 메서드가 깨어나 리턴되고 난 후에 조건이 만족되었다고 간주하면 안되고, 다시 조건을 확인해야 한다.
* 결국, wait 메서드를 반복문 안에 넣어 사용해야 하며 반복할 때 마다 계속해서 조건 서술어를 확인해야 한다.

```java
void stateDependentMethod() throws InterruptedException {
    synchronized(lock) {
        while (!conditionPredicate())
            lock.wait();
        // 객체가 원하는 상태에 도달하면 이 위치에 오게 된다.
    }
}
```

* 조건 서술어를 확인한 후 실제 작업을 시작하고 끝날 때까지 락을 해제하면 안된다.

### 놓친 신호

* 특정 스레드가 이미 조건을 만족하는 상황에서 조건 서술어를 제대로 확인하지 못해 대기 상태에 들어가는 상황을 의미한다.
* 보통 스레드에 대한 알람이 일시적이므로 이런 문제가 발생한다.&#x20;
* 스레드가 조건 큐에 신호를 보내 이미 지나간 후에 다른 스레드가 조건 큐에서 대기한다면, 해당 스레드는 영원히 대기 상태에 머무를 수 있다.
* notifyAll 대신 notify 메서드를 사용할 경우 놓친 신호와 유사한 문제가 생길 가능성이 있다. 깨어난 스레드가 대기 상태에 들어갔을 때의 조건이 현재 만족된 조건과 다를 수 있기 때문이다. 즉, 단일 조건에 따른 대기 상태가 아닌 경우 이러한 문제가 발생하게 된다.
* notify 메서드는 단일 조건에 따른 대기 상태에서 깨우는 경우, 한 번에 하나의 스레드만이 작업을 처리하는 경우에는 사용해도 좋다. 이외의 경우에는 notifyAll 메서드를 사용하는게 안전하다.

### 알림

* 대기의 반대되는 부분은 알림이다. 대기 상태에서 스레드를 빠져나오게 하려면 알림이 필요하다.
* 상태 변수가 변경될 때마다 알림을 보낼 수도 있지만, 조건 서술어가 만족할만한 상태가 되었을 때만 알람을 보내도록 하면 최적화할 수 있다.
* 상태 기반으로 동작하는 클래스는 하위 클래스에게 대기와 알림 구조를 공개하고 문서로 남기거나, 하위 클래스에서 대기와 알림 구조에 접근 불가능하도록 제한해야 한다.
* 하위 클래스에서 새로 추가된 메서드로 인해 조건이 추가되는 경우, 기존에 단일 조건 서술어를 사용하였기 때문에 단일 알림 방법을 사용했던 것을 notifyAll로 변경해야 할 수 있다.
* 조건 큐는 클래스 내부에 캡슐화하여 클래스 상속 구조 외부에서는 접근하지 못하도록 막아야 한다.
* 조건 큐에 대한 대기와 알림 규칙을 추측한 상태에서 별도로 호출하게 되면 의도치 않은 동작이 발생하여 문제가 생길 수 있다.

## 명시적인 조건 객체

* 암묵적인 락과 암묵적인 조건 큐를 같이 사용하게 되면 하나의 조건 큐를 여러 스레드가 다양한 조건을 기준으로 삼아 대기 상태에 들어갈 수 있다. 또한 조건 큐 객체를 스레드에게 노출시키게 되며, 단일 대기 조건을 만족시키기가 불가능하다.

#### Condition

* 자바에서는 암묵적인 조건 큐를 일반화한 명시적인 조건 객체인 Condition 클래스를 제공한다.
* Condition 클래스는 하나의 락에 여러 조건으로 대기하게 할 수 있으며, 인터럽트 반응, 데드라인을 정해둔 대기 상태 등 다양한 기능을 제공한다.
* Lock 객체로부터 Condition 객체를 생성 가능하며, 동일한 공정성 레벨을 받게 된다.
* 대기 조건들을 각각의 조건 큐로 나누어 대기하도록 하면 단일 알람 조건을 쉽게 만족시킬 수 있고 이 덕분에 signalAll 대신 효율적인 signal 메서드를 통해 대기중인 스레드를 깨울 수 있다.
* 조건에 관한 모든 변수는 Lock으로 보호되어 있어야 하며 조건을 확인하거나 await, signal 메서드를 호출하는 시점에는 Lock을 확보한 상태여야 한다.

#### 예시

* 다음은 버퍼에 값을 추가할 때 Lock과 Condition을 사용하는 예시이다. 버퍼에 값이 꽉찼으면 notFull 신호가 올 때 까지 락을 잠시 풀고 대기한다. 아이템을 추가할 때 take 메서드에서 마찬가지로 버퍼에 값이 생길 때까지 기다리는 notEmpty 신호를 기다리고 있을 수 있으니, 아이템을 추가한 후 notEmpty 신호를 전달한다.
* 이 때 Condition 객체를 여러 개 두어 단일 조건 대기 상태를 만족시키면

```java
Lock lock = new ReentrantLock();
Condition notFull = lock.newCondition();
Condition notEmpty = lock.newCondition();

public void put(Object x) throws InterruptedException {
    lock.lock();
    try {
        while (count == items.length) 
            notFull.await(); // 버퍼가 꽉 차면 생산자 대기
        
        // 아이템 추가 로직
        
        notEmpty.signal(); // 아이템이 생겼으니 소비자에게 알림
    } finally {
        lock.unlock();
    }
}
```

## AbstractQueuedSynchronizer

* 동기화 클래스를 작성할 때 필요한 다양한 기본 구현들을 제공한다.
* AQS 기반으로 만들어진 클래스는 대기 상태에 들어갈 수 있는 지점이 하나뿐이므로 컨텍스트 스위칭 부하를 줄이고 성능을 높일 수 있다.

#### 확보/해제 연산

* 가장 기본이 되는 연산은 acquire(확보)와 release(해제)이다.
* 확보 연산
  * 항상 원하는 상태가 될 때까지 대기할 가능성이 있다.
  * 예를 들어 CountDownLatch에서는 래치가 완료될 때 까지 기다려야 확보 가능하고, FutureTask에서는 작업이 완료될 때 까지 기다려야 확보 가능하다.
  * 확보 연산은 1. 동기화 클래스에서 확보 연산을 허용하는 상태인지 확인하고 2. 동기화 클래스 내부의 상태를 업데이트하는 과정으로 이뤄진다.
* 해제 연산
  * 대기 상태에 들어가지 않으며, 확보 연산에서 대기 중인 스레드를 풀어주게 된다.

#### 상태 정보 관리

* 단일 int 변수 기반의 상태 정보를 관리할 수 있는 getState/setState/compareAndSetState 메서드를 제공한다.
  * 예를 들어 ReentrantLock 클래스에서는 이 변수를 통해 스레드가 락을 몇 번 확보했는지 관리하고, Semaphore 클래스는 남아있는 퍼밋의 개수를 관리한다.

#### 구현해보기

* 배타적인 확보 기능을 제공하는 동기화 클래스라면 tryAcquire, tryRelease, isHeldExclusively 등의 메서드를 지원해야 한다.
* 배타적이지 않은 확보 기능을 제공하는 동기화 클래스는 tryAcquireShared, tryReleaseShared 메서드를 제공해야 한다.
* 해제 연산 메서드가 완료되고 확보 연산에서 대기하는 스레드를 풀어줄 수 있는 상태라면, true를 반환해야 한다.
* 다음은 AQS를 위임 기법으로 사용하여 동기화 클래스를 만든 간단한 예시이다.
  * tryReleaseShared에서 true를 반환해야 await 메서드로 대기하던 스레드들을 깨울 수 있다.

```java
@ThreadSafe
public class OneShotLatch {
       private final Sync sync = new Sync();
       
       public void signal() { sync.releaseShared(0); }
       
       public void await() throws InterruptedException {
              sync.acquireSharedInterruptibly(0);
       }
       
       private class Sync extends AbstractQueuedSynchronizer {
             protected int tryAcquireShared(int ignored) {
                    // 래치가 열려 있는 상태(state==1)라면 성공, 아니면 실패
                    return (getState()==1) ? 1 : -1;
             }
             
             protected boolean tryReleaseShared(int ignored) {
                    setState(1); // 래치가 열렸다.
                    return true; // 다른 스레드에서 확보 연산에 성공할 가능성이 있다.
             }
       }
}
```
